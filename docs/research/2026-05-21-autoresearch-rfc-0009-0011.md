## 1. Executive Summary

**RFC 0008 Phase E (federation deps + lockfile).** The Go module system is the
closest production proof-of-concept for what Mind.toml Phase E needs: a
`go.sum`-style dual-file scheme separates human-readable dep declarations
(`go.mod` / `Mind.toml`) from machine-owned content hashes (`go.sum` /
`Mind.lock`). Go achieves 99.7% lockfile adoption precisely because the lock is
generated and enforced by default with zero user ceremony. The top recommendation
for Phase E is to adopt this two-file split verbatim: `Mind.toml` holds
`rev`-pinned git URLs; `Mind.lock` records SHA256 of the fetched tree, written
on first fetch, verified on every subsequent fetch, committed to VCS.

**RFC 0009 (federation-first package layer).** Deno's 2024 post-mortem on HTTP
imports is the clearest documented failure of URL-only identity without a
registry layer. The lesson is not "add a central registry" but "add a
content-addressed proxy protocol with a defined mirror API so any server can
serve it." The top recommendation is to define a lightweight Mind Package Protocol
(MPP): a static HTTP layout over a git host so any mirror can serve it — the
same design Go's module proxy protocol uses — and make the official
`pkg.mindlang.dev` mirror just one conforming node.

**RFC 0010 part A (C ABI in pure MIND).** Zig 0.14's redesign of
`CallingConvention` into an exhaustive tagged union (`.x86_64_sysv`,
`.arm_aapcs`, `.win64`, etc.) is the best prior art for expressing platform
ABIs without macros. The top recommendation is to adopt the same shape: a
`callconv` attribute whose values are an enumerated set of platform-ABI names
compiled into the MIND type system, with `callconv(.c)` as the default alias for
the target's C ABI — identical in spirit to Zig's `callconv(.c)`.

**RFC 0010 part B (memory safety model).** The two most relevant 2025 findings
are Spegion (ECOOP 2025) — implicit non-lexical regions without substructural
types, enforced via an effect system — and Vale's generational-references +
optional region borrow-check hybrid. For MIND, a three-tier invariant model is
recommended: (1) stack-allocated values are always safe by construction; (2)
heap pointers carry a generation tag at negligible cost and are checked on
dereference; (3) a lightweight `region` annotation demarcates arenas that free
together, eliminating the tag cost inside the region. This avoids the borrow
checker's ergonomic tax while maintaining the byte-identity / determinism
properties MIND requires.

**RFC 0011 (async / structured concurrency).** Zig's new `Io` interface pattern —
where I/O strategy (blocking, thread-pool, event-driven) is injected as a
parameter rather than baked into function signatures — is the only 2025 design
that demonstrably eliminates function coloring without adding a coloring effect
of its own. C++26 `std::execution` (P2300, adopted June 2024) provides the
complementary algebra: senders and receivers as composable async values.
The top recommendation for RFC 0011 is to adopt a `mind.io.Scheduler` interface
injected via MIND's context system — analogous to Zig's `Io` parameter and
Jai's context allocator — so library code is agnostic to execution strategy and
deterministic replay is achievable by swapping schedulers.

## 2. RFC 0008 Phase E — Federation Deps + Content-Hash Lockfile

### 2.1 Findings

#### Finding E-1: Go's two-file model is the empirically best lockfile UX

The paper "The Design Space of Lockfiles Across Package Managers"
(arXiv:2505.04834, 2025) studied npm, Cargo, Go, Poetry, Gradle, and others.
Go achieves 99.7% lockfile adoption; Gradle achieves 0.9%. The decisive factor
is whether the lock is generated and enforced **by default** with no flags.
Go's `go.sum` contains only hash lines, is diff-friendly, and is automatically
updated when `go get` runs. npm's lockfile includes extraneous metadata (funding
info, OS requirements) that makes diffs expensive.

Cargo ignores its own lockfile for library crates unless `-locked` is passed —
a documented source of "it built on my machine" failures. Mind.toml must not
repeat this: `mindc build` must fail if `Mind.lock` is absent and no `--update`
flag is given, matching Go's default enforcement posture.

#### Finding E-2: Content hash over the fetched tree, not over the git object

Nix flakes store `narHash` (hash of the unpacked directory tree) separately from
the git revision. This matters because the same git commit can produce different
tarballs depending on archive flags. MIND's `Mind.lock` should hash the
**unpacked source tree** (analogous to Nix's NAR hash), not the git object ID,
to maintain byte-identity across mirror transitions.

#### Finding E-3: GOPROXY's mirror protocol is a proven federation blueprint

Go's module proxy protocol is a static HTTP layout:
`{base}/{module}/@v/{version}.info`, `.mod`, `.zip`. Any HTTP server can
implement it. GOPROXY falls through a comma-separated list:
`GOPROXY=https://org-mirror.example.com,direct`. This trivially handles
enterprise airgap and private-module scenarios without a custom registry server.
The Mind analogue is a `MINDENV_PROXY` environment variable with the same
fall-through semantics.

#### Finding E-4: SHA256 is the right hash primitive; Blake3 is the upgrade path

Cargo uses SHA256. Go uses SHA256. Nix uses SHA256 (transitioning to SHA512 for
NARs). The 2024 supply-chain attack literature (SLSA blog, 2024-08) does not
identify SHA256 collision risk as a practical attack vector — the attack surface
is name resolution, not hash strength. Phase E should use SHA256. A future phase
can add Blake3 as an opt-in without breaking the lockfile format by adding a
`hash-algo` field.

#### Finding E-5: TOFU-then-pin is the right trust model for git deps

A git URL with a pinned `rev` is already content-addressed once the rev is
known. The first fetch establishes a `Mind.lock` entry; all subsequent fetches
verify against it. This is exactly TOFU (Trust On First Use) which is the
standard model for SSH host keys and Go module checksums. The only security
requirement is that the first fetch is over an authenticated transport (HTTPS).

### 2.2 Recommendations for Phase E

**E-R1. Two-file scheme: `Mind.toml` declares intent, `Mind.lock` owns truth.**
`Mind.lock` is a machine-written, VCS-committed JSON file keyed by
`"<git-url>#<rev>"` with fields `tree_sha256` (SHA256 of the unpacked source
tree), `fetch_url` (canonical URL used), and `timestamp_utc` (ISO 8601 fetch
time for audit purposes). Unknown fields in the lock entry are an error on read,
not silently ignored.

**E-R2. Default-enforce: `mindc build` fails without `Mind.lock` unless
`--update-lock` is passed.** Lock update is an explicit user action, never
implicit on a normal build. This matches Go's strict default and is the single
biggest driver of adoption and security.

**E-R3. Implement `MINDENV_PROXY` with Go-style fall-through list.** The default
value is `direct` (fetch from the git URL in `Mind.toml`). An organization mirror
prepends its URL. An airgap deployment sets `MINDENV_PROXY=https://internal/,off`
where `off` means "error rather than fall through to direct." This requires zero
server-side infrastructure for the common case, and a simple static file server
for the mirror case.

## 3. RFC 0009 — Federation-First Package Layer

### 3.1 Findings

#### Finding 09-1: Deno's HTTP-import experiment documents the canonical failure modes

Deno's 2024 retrospective "What We Got Wrong About HTTP Imports" is the most
detailed post-mortem of URL-only package identity. The documented failures are:

- **Reliability**: dependencies on personal servers brake CI; availability depends
  on the least-reliable node in the dep graph.
- **Version ambiguity**: without semver enforcement, projects accumulate multiple
  variants of the same URL.
- **Discoverability**: no index, no search, no typosquatting defense.

Deno's fix was not to abandon URL identity but to add a registry layer (JSR) that
**still uses HTTP imports underneath** with semantic versioning enforcement on top.
MIND's RFC 0009 is already one step ahead: it starts with git-URL identity
(content-addressed by commit hash) and needs to add discoverability + semver
layering, not the other way around.

#### Finding 09-2: The dependency-confusion attack is structurally absent when identity is a git URL

Dependency confusion (an attacker publishes a public package with the same name
and a higher version than a private package) requires a name-based resolution
system. When package identity is a full git URL pinned to a `rev`, there is no
namespace to confuse. This is a structural security advantage of the federation
model over crates.io / npm-style registries. RFC 0009 should preserve this
invariant: a package's canonical identity is its git URL, not a short name.
Short names (aliases) are a local `Mind.toml` convenience, never used in the
lock file.

#### Finding 09-3: Nix flakes show how to add a lock layer above git identity

Nix flakes use `url + rev + narHash` as the canonical triple. The `flake.lock`
stores this triple for every transitive dep. The `follows` attribute allows
multiple deps to pin to the same version of a shared dep — solving the
diamond-dependency problem without a global semver resolver. MIND can adopt
the same mechanism: `Mind.lock` records the triple; a `uses = other-dep/dep-name`
field in `Mind.toml` expresses a follows-override.

#### Finding 09-4: JSR's open-source registry design is the right model for an optional index layer

JSR is MIT-licensed, operates over HTTP, enforces semver, auto-generates
documentation from source. It is explicitly designed to be self-hostable. Its
package publishing protocol is a set of well-defined REST endpoints over a simple
schema. RFC 0009 can specify a "MIND Package Index Protocol" (MPIP) as an
optional discovery layer — the same content is fetchable by git URL without the
index. The index adds search, semver ranges, and provenance metadata; it does not
change what is fetched.

#### Finding 09-5: Go's `GOPRIVATE` env var pattern solves the public/private split cleanly

The `GOPRIVATE=gitlab.example.com/internal/*` glob causes the Go toolchain to
bypass the public proxy and checksum DB for matching paths. MIND needs an
identical mechanism: `MINDENV_PRIVATE=<glob>` causes `mindc` to bypass any
configured proxy and MPIP index for matching git URLs, fetching directly and
recording only in the local `Mind.lock`.

### 3.2 Recommendations for RFC 0009

**09-R1. Package identity = git URL. Short names = local aliases only.**
The wire format in `Mind.lock` always uses the full git URL + rev + tree_sha256
triple. `Mind.toml` `[dependencies]` may use short aliases for ergonomics but
the lock resolves them to full triples immediately.

**09-R2. Define MPIP as a static HTTP layout over a git host.**
A conforming MPIP server responds to:
- `GET /index/<encoded-url>/versions.json` — available semver tags
- `GET /index/<encoded-url>/<rev>/manifest.json` — Mind.toml contents
- `GET /index/<encoded-url>/<rev>/tree.sha256` — pre-computed tree hash

Any git host with a CI artifact pipeline (GitHub Actions, Gitea CI) can auto-publish
MPIP metadata without a dedicated registry server.

**09-R3. Implement semver range resolution in RFC 0009, not Phase E.**
Phase E pins to a specific `rev`. RFC 0009 adds `version = "^0.3"` which the
MPIP index resolves to the highest matching semver tag, then pins the resulting
`rev` in `Mind.lock`. This layering ensures Phase E code keeps working unchanged
when RFC 0009 lands.

**09-R4. Mirror-first, index-optional.** An `MINDENV_PROXY` mirror that satisfies
MPIP is a first-class citizen. The official `pkg.mindlang.dev` mirror is one
implementation. Any organisation can run its own. An airgap deployment never
contacts `pkg.mindlang.dev` and still builds correctly.

## 4. RFC 0010 Part A — C ABI in Pure MIND

### 4.1 Findings

#### Finding 10A-1: Zig 0.14's `CallingConvention` tagged union is the right shape

Zig 0.14.0 (released early 2025) redesigned `CallingConvention` from an enum
into an exhaustive tagged union with entries like `.x86_64_sysv`,
`.x86_64_win64`, `.arm_aapcs_vfp`, `.aarch64_aapcs`, `.riscv64_interrupt`, and
a target-aliased `.c` for the default C ABI of the compilation target.
`CommonOptions` payload allows overriding stack alignment per-callsite.
Deprecation of the old single-name forms (`.C`, `.Stdcall`) with clear migration
paths shows how to evolve a calling-convention model without breaking existing
code.

MIND currently has no first-class syntax for calling conventions beyond the
implicit C ABI used by `pub fn` exports (RFC 0002). RFC 0010 needs to make the
ABI explicit and exhaustive, matching Zig's approach.

#### Finding 10A-2: Rust 2024 Edition's `unsafe extern "C"` safety split is a directly adoptable pattern

In Rust Edition 2024, `extern` blocks must be marked `unsafe`. Individual
symbols inside the block can then be declared `safe` (no `unsafe` block needed at
call site) or `unsafe` (require `unsafe` at call site). This per-symbol safety
annotation resolves a long-standing tension: most `libc` math functions are
genuinely safe to call (they do not require pointer validity invariants from the
caller), but `memcpy` is not. The distinction is load-bearing for ergonomics.

MIND's `extern` block syntax should adopt the same split: the block is unsafe by
declaration, but individual symbol bindings carry an explicit `safe` or `unsafe`
marker that propagates to call sites.

#### Finding 10A-3: The MLIR C API is the correct FFI entry point for RFC 0010's libMLIR integration

MLIR exposes a stable C API (`libMLIRCAPIIR`) specifically intended as the FFI
entry point for non-C++ languages. The Swift MLIR binding work (Duan, 2024)
demonstrates the pattern: a thin `shim.h` + module map, then call C functions
directly. The pain points are (1) finding the correct library name
(`MLIRCAPIIR` not `libMLIR`) and (2) struct layout for opaque handle types.
Both are manageable with a code-generated binding layer.

For MIND, the recommendation is to generate the C ABI declarations for MLIR from
the MLIR C API headers via a `mind bindgen` subcommand (analogous to
`rust-bindgen`), producing a `mlir.mind` module that declares all MLIR handles
as opaque pointer types with explicit `callconv(.c)` annotations.

#### Finding 10A-4: System V x86_64 struct-passing rules are the most important to get right

The System V AMD64 ABI classifies struct fields into "eightbytes." A struct
fitting within two eightbytes of INTEGER or SSE class is passed in registers; a
larger struct or one containing unaligned fields is passed via a hidden pointer
in `rdi`. Getting this wrong silently produces incorrect results — the caller
and callee disagree on where arguments live. Win64 has a different rule: structs
larger than 8 bytes are always passed by pointer. AArch64 AAPCS has a third rule:
HFA (homogeneous floating-point aggregate) structs up to four members pass in
SIMD registers.

MIND's FFI layer must encode these classification rules for each supported
platform rather than delegating them entirely to LLVM. The correct approach is to
express struct arguments in MIND's IR as either by-value (for small INTEGER-class
structs) or as `*T` with a `sret` attribute (for structs passed by hidden
pointer), matching LLVM's own representation after ABI lowering.

#### Finding 10A-5: Jai's `#foreign` approach is minimal and ergonomic

Jonathan Blow's Jai uses `#foreign <lib>` on a function declaration with no
separate `extern` block syntax. The declaration itself is the canonical
description of the foreign symbol; the compiler infers the C ABI automatically.
Jai even demangles C++ names on request. The ergonomic lesson for MIND: the
`extern` annotation should attach to the function declaration directly, not
require a separate block, for the common case of individual symbol imports.
An `extern` block remains available for bulk imports from generated bindings.

### 4.2 Recommendations for RFC 0010 Part A

**10A-R1. Add a `callconv` attribute with an exhaustive ABI enum.**
Values: `c` (alias for target's C ABI), `sysv` (x86_64 System V),
`win64` (x86_64 Microsoft), `aapcs` (AArch64), `naked` (no prologue/epilogue).
Additional platform-specific values are addable without breaking changes by
extending the enum. Syntax: `fn foo() callconv(.sysv) -> i32`.

**10A-R2. Adopt Rust 2024's unsafe/safe split inside extern blocks.**
Every `extern` block is implicitly `unsafe`. Each symbol binding within it is
annotated `safe` or `unsafe`. A symbol annotated `safe` can be called without
an `unsafe` expression; a symbol annotated `unsafe` requires one. The default
for unannotated symbols is `unsafe` — safer to be conservative.

**10A-R3. Implement `mind bindgen` to auto-generate MIND FFI declarations from
C headers.** Target MLIR C API headers first. The generated module is checked in
to the repository and not regenerated on normal builds, matching how
`rust-bindgen`-generated files are typically handled.

## 5. RFC 0010 Part B — Memory Safety Model

### 5.1 Findings

#### Finding 10B-1: Spegion (ECOOP 2025) — implicit non-lexical regions without substructural types

Spegion (Hughes, Vollmer, Batty; ECOOP 2025, arXiv:2506.02182) is the most
directly relevant recent paper. The core contribution is that region-based memory
safety can be enforced with an **effect system** rather than substructural types
(linear/affine). Effect annotations on function types record which regions the
function can allocate into and read from, without requiring the programmer to
annotate every borrow site. Regions may be split into sub-regions (the "splittable"
in the name), enabling fine-grained control that lexical scoping cannot express.

For MIND, the practical takeaway is that a `region` keyword that introduces a
lexically-scoped arena is not sufficient — the RFC 0010 model needs non-lexical
region lifetimes (a region created in one function, passed to another, freed in
a third) which requires effect-system tracking at the type level.

#### Finding 10B-2: Vale's generational references eliminate use-after-free without a borrow checker

Vale's generational references (Wesnoski, 2020–2024) attach an 8-byte generation
counter to each allocation. A non-owning reference carries the generation number
at creation time; every dereference asserts the current generation matches. The
overhead is one integer comparison per dereference of a non-owning reference.
Owning pointers never pay this cost. When a region is active, non-owning
references into it can be made free (zero-cost) because the entire region is
known to be live — this is Vale's planned hybrid-generational model, analogous
to what Spegion formalises.

For MIND, this maps directly: inside a `region` block, all pointer dereferences
are free (the region is live by construction); outside, non-owning pointers carry
generation tags with a single-comparison check. This satisfies the governance
substrate requirement: deterministic behavior, no garbage collection pauses.

#### Finding 10B-3: Cyclone's retrospective — regions work well for stack and arenas, poorly for event loops

Cyclone (Grossman, Morrisett, et al., 2002–2006; ISMM 2004 retrospective paper)
established both what regions can do and what they cannot. The finding that LIFO
arenas "are not suited to computations such as server and event loops" is directly
relevant to RFC 0011: any server-loop or async task that spans multiple stack
frames cannot use lexical region lifetimes for its allocations. Non-lexical
regions (Spegion) or generational references (Vale) are the two viable paths for
async code.

#### Finding 10B-4: RefinedRust (PLDI 2024) shows that ownership + refinement types enables formal verification

RefinedRust (Gäher, Sammler, Jung, Krebbers, Dreyer; PLDI 2024) extends Rust's
ownership type system with refinement types and Iris separation logic to prove
functional correctness of unsafe Rust code. The key insight is that ownership
already gives the frame condition for free — a function that takes exclusive
ownership of a data structure can only affect it, not anything else. This is
relevant to MIND's governance-substrate positioning: an ownership-based memory
model is a prerequisite for the kind of auditability and evidence-chain properties
MIND advertises. A GC model would undermine these properties.

#### Finding 10B-5: Hylo's mutable value semantics (MVS) avoids borrow-checker ergonomic tax

Hylo (formerly Val; Abrahams, Poetter; CppOnSea 2024, BLDL 2024) bases its
memory safety on mutable value semantics (MVS): values are logically independent
copies; in-place mutation is an implementation optimisation, not a semantic
primitive. The four parameter-passing modes (`let`, `inout`, `sink`, `set`)
replace Rust's `&`, `&mut`, ownership-move, and default-init. The advantage over
Rust's borrow checker is that `inout` does not require the caller to annotate at
the call site — the callee's signature is sufficient. Hylo has a working compiler
as of late 2024 and is no longer experimental.

For MIND, MVS is a direct fit: Q16.16 values are already value types (they are
fixed-width integers), and MIND's current semantics treat everything as by-value
by default. The RFC 0010 memory model can formalise this as MVS with explicit
`inout` passing for mutation.

#### Finding 10B-6: ATS demonstrates that dependent types + linear types is too steep a learning curve for adoption

ATS (Xanadu/ATS3 still in development as of 2024) provides linear types plus
dependent types plus proof terms in one language. Despite offering the strongest
safety guarantees of any systems language, fewer than 200 GitHub repositories
contain ATS code as of 2024. The learning curve is the identified barrier.
MIND must not repeat this: the memory safety model for RFC 0010 should express
99% of use cases without proof terms, with an optional escape hatch for
verification-level assurance.

### 5.2 Recommendations for RFC 0010 Part B

**10B-R1. Adopt a three-tier memory model: stack (free), region (free inside,
generation-checked outside), heap (always generation-checked).**
- Stack-allocated values: no cost, no annotation.
- Region allocations inside a `region` block: no cost, freed when the block exits.
- Non-owning references outside their region's lifetime: carry an 8-byte
  generation tag, one comparison per dereference.
- Heap allocations (no enclosing region): same as non-owning references.

**10B-R2. Use MVS parameter-passing conventions for the safety interface.**
Four modes: `let` (immutable), `inout` (mutable in-place), `own` (ownership
transfer), `set` (callee-initialises). These replace ad-hoc pointer patterns.
Mutation outside `inout` is a compile-time error in safe code.

**10B-R3. Non-lexical region lifetimes via effect annotations.** A function's
type signature can include region effects: `fn foo[r: Region](x: &r T) -> &r U`.
The effect system (inspired by Spegion and Koka) tracks which regions a function
can read/allocate into without requiring per-site borrow annotations.

## 6. RFC 0011 — Async / Structured Concurrency

### 6.1 Findings

#### Finding 11-1: Zig's Io interface is the only 2025 design that eliminates function coloring

The function-coloring problem is that `async` functions can call `sync` functions
but not vice versa, requiring library code to be duplicated in sync and async
flavors. Zig's new async I/O design (Kristoff, blog 2025; Zig 0.16 release notes)
solves this by making the execution strategy an injected parameter (`Io`) rather
than a function signature attribute. A function that accepts an `Io` parameter
can perform I/O in any execution mode; a function that does not accept `Io` is
purely computational and callable from any context. Three concrete backends are
planned: blocking, thread-pool, and io_uring/green-threads.

The Io injection pattern is isomorphic to MIND's existing context system (Jai-
style allocator injection). This is a natural fit: add a `mind.io.Scheduler`
interface to the context, and functions that perform I/O accept it implicitly.

#### Finding 11-2: C++26 `std::execution` (P2300) provides a proven algebra for composable async values

P2300 was adopted into C++26 at the St. Louis WG21 plenary, June 2024. The
central abstractions are senders (async values), receivers (callbacks that handle
value/error/done signals), and schedulers (execution contexts). Senders compose
via algorithms (`then`, `when_all`, `on`, `just_stopped`). The design is
explicitly structured: a sender represents a computation with bounded lifetime
that runs to completion, reports success/failure/cancellation.

For MIND, the composable algebra of senders maps cleanly onto RFC 0011's async
expression needs: an `async` block in MIND should desugar to a sender; `await`
should desugar to a receiver connection; `Scheduler` should be the injected
context. The structured-concurrency guarantees (all spawned tasks complete before
the enclosing scope exits) are preserved by the sender/receiver lifetime model.

#### Finding 11-3: Swift 6's region-based isolation shows how to make actor isolation ergonomic

Swift 6 (WWDC 2024) introduced SE-0414 region-based isolation as a replacement
for the blunt `Sendable` requirement. Instead of requiring every type to be
`Sendable` to cross actor boundaries, the compiler tracks *regions* of values and
proves that no two concurrent tasks access the same region. This eliminates
false-positive `Sendable` errors — the dominant complaint about Swift's strict
concurrency — while preserving data-race safety.

For MIND, region-based isolation maps naturally onto the three-tier memory model
from §5: a value in a `region` block is already isolated; the async scheduler
can move ownership of a region between tasks without any additional annotation.

#### Finding 11-4: Swift 6.2 reduced friction proves that strict concurrency is only deployable if escape hatches are easy

Swift 6.2's main change was making `@MainActor` the default for most code,
drastically reducing the annotation burden for existing codebases. The lesson is
that strict concurrency guarantees are only adopted at scale when the **default
path** (i.e., adding no annotations) produces code that is already safe. For
MIND, the corollary is: the default async mode should be single-task (no
concurrency), and concurrency should require an explicit `concurrent` or
`task_group` call. Safety is free by default; concurrency is opt-in.

#### Finding 11-5: Erlang/OTP supervision trees are the proven model for fault-tolerant task hierarchies

OTP's supervision tree predates the term "structured concurrency" by two decades
and implements the same invariant: a supervisor owns its children; if a child
dies, the supervisor restarts it per a declared strategy. The Gleam language
(v1.0, March 2024) brings a typed wrapper around this model, and NVLang
(arXiv:2512.05224, December 2025) formalises typed `Pid[T]` process identifiers
that encode the message protocol a process expects.

For MIND, a supervision tree primitive in RFC 0011 provides fault tolerance for
long-running governance pipelines — the core use case of MIND's governance
substrate. A `mind.task.Supervisor` type with `one_for_one`, `one_for_all`, and
`rest_for_one` restart strategies directly adopts the OTP vocabulary.

#### Finding 11-6: Zig's async removal retrospective identifies four concrete pitfalls

Zig's 0.12 release notes document four unsolved problems with the old stackless
coroutine approach: (1) LLVM cannot optimize stackless coroutines as well as
regular functions; (2) third-party debuggers cannot debug them; (3) the
cancellation problem — how to safely cancel a suspended coroutine; and (4) async
function pointer types prevent the compiler from knowing stack size. The new Io
interface approach deliberately avoids all four by keeping functions as regular
functions and making the scheduler, not the function, responsible for suspension.

#### Finding 11-7: OCaml 5's algebraic effect handlers impose only 1% overhead on non-effectful code

The OCaml multicore team measured a 1% mean overhead across a comprehensive macro
benchmark suite for code that does not use effect handlers, while code that does
use them performs on par with manually optimized monadic concurrency libraries.
This demonstrates that effect-based concurrency can be a zero-regret addition to
a language without a global performance penalty.

#### Finding 11-8: Koka's ndet effect tracks non-determinism through the type system

Koka (v3.1.0, February 2024) tracks non-determinism as an explicit effect `ndet`.
A function with type `() -> <ndet> int` is non-deterministic; one with type
`() -> pure int` is not. This is directly applicable to MIND's governance
substrate: a governance rule that invokes an LLM is `<ndet>`; one that does pure
Q16.16 arithmetic is `pure`. The effect type enforces that non-deterministic
functions cannot be called from deterministic governance pipelines without an
explicit `handle` block that either samples once (pinning the result) or produces
a distribution.

For MIND's RFC 0011, this means async effects and ndet effects are distinct: an
async function can still be deterministic if its scheduler is deterministic (e.g.,
a single-threaded replay scheduler).

#### Finding 11-9: Deterministic record & replay requires explicit modeling of all nondeterminism sources

The Programming Journal 2022 paper on concurrency-model-agnostic record & replay
and the 1805.06267 arxiv paper on deterministic R&R for actor languages both
conclude the same thing: deterministic replay is achievable for any concurrency
model so long as all nondeterminism sources (I/O, scheduling, random, time) are
captured in a trace. MIND's governance-substrate positioning means RFC 0011
should include a `ReplayScheduler` that replays a captured trace, making any
MIND program's async execution deterministically reproducible.

### 6.2 Recommendations for RFC 0011

**11-R1. Scheduler injection via context, not function coloring.**
Add `mind.io.Scheduler` to the MIND context object. Functions that perform I/O
or spawn tasks accept it implicitly. Pure computational functions do not. No
`async`/`sync` function-signature dichotomy.

**11-R2. Sender/receiver algebra for async composition.**
An `async { ... }` block evaluates to a `Sender[T]`. `await sender` in a
scheduled context consumes the sender and yields `T`. Composition operators:
`then`, `when_all` (concurrent join), `race` (first-completed). These desugar
to sender chains, preserving structured lifetime.

**11-R3. Supervision trees as first-class task hierarchy.**
`mind.task.Supervisor` with OTP-style strategies. A supervised task that panics
is caught by its supervisor; the panic does not propagate up the stack. This is
load-bearing for governance pipelines that run untrusted plugin code.

**11-R4. Expose `ndet` as an explicit effect type for non-determinism.**
Functions that call LLM APIs, read from `/dev/urandom`, or depend on wall-clock
time are `<ndet>`. A governance rule that must be deterministic can only call
`<ndet>` functions inside an explicit `commit` block that pins the result and
records it in the evidence chain.

**11-R5. Ship a `ReplayScheduler` alongside the default scheduler.**
A `ReplayScheduler` takes a trace file and replays scheduling decisions
deterministically. Any test that exercises async governance logic can be run
under `ReplayScheduler` for perfect reproducibility. This is the RFC 0011
deliverable that most directly advances MIND's governance-substrate positioning.

## 7. Cross-Cutting Concerns

### 7.1 Effect systems are the right cross-cutting abstraction for §5–§6

Both the memory safety model (§5, Spegion) and the async concurrency model (§6,
Koka/OCaml 5) converge on effect systems as the mechanism for tracking
computational properties through the type system. Region effects, I/O effects,
and ndet effects are all instances of the same abstraction. MIND should not
implement three separate tracking mechanisms; it should implement one effect
system parameterised over effect labels.

Concretely: a function's type in MIND should carry an optional effect row
`<io, ndet, region:r>` in the same syntactic position, lowered to LLVM function
attributes for the optimizer's benefit.

### 7.2 The determinism guarantee must be a first-class property, not a convention

MIND's governance substrate requires that governance rules produce the same output
given the same input. Both the memory safety model and the async model must
cooperate to maintain this. The mechanism is the `ndet` effect: any function that
can produce different outputs on the same input must carry `ndet`; governance
rules that must be deterministic are `ndet`-free by the type system.

This is the same guarantee that Starlark (Bazel's configuration language) achieves
by making it a deterministic subset of Python — no file I/O, no random, no time
access. MIND should offer Starlark-style determinism as a verifiable property
(via the `ndet`-free effect row) rather than a convention.

### 7.3 Context injection is the unifying deployment primitive

Jai's context-allocator, Zig's Io-parameter, Swift's actor isolation, and MIND's
existing governance context are all instances of the same pattern: behavior that
varies by deployment (allocator strategy, I/O backend, randomness source,
scheduler) is injected via a well-known context object rather than baked into
function signatures or global state. MIND already has the context object. The
right move for RFCs 0010–0011 is to expand it, not add parallel mechanisms.

### 7.4 Content addressing is the right primitive for both lockfiles and caching

RFC 0008 Phase F (incremental compilation cache) and Phase E (lockfile) both
use SHA256 content hashes as keys. The Bazel/Turborepo literature (2024–2025)
confirms that content-addressed caches are simpler to reason about, compose
better across machines, and are immune to clock-skew invalidation bugs that plague
mtime-based caches. MIND's `~/.mindenv/cache` should be a flat content-addressed
store keyed by `sha256(source_tree + flags + dep_hashes)` — identical in
structure to Nix's store. Objects in the store are immutable; garbage collection
is by explicit `mindc cache clean --older-than=30d`.

## 8. Anti-Patterns to Avoid

### AP-1: HTTP-URL-only package identity without a hash layer (Deno's original HTTP imports, 2018-2023)

Deno's experience is definitive: a URL is not a content address. The same URL
can serve different content at different times (the server operator may update it,
the CDN may cache a stale version, or the domain can be taken over). RFC 0009
must not allow a package dep whose only identity is a URL without a pinned hash.
The minimum anchor is `git URL + rev + tree_sha256`.

### AP-2: Optional lockfile enforcement (Cargo's behavior for library crates)

Cargo silently regenerates `Cargo.lock` for library crates instead of failing
when the lock is stale. This means a library author's CI may use different dep
versions than a downstream user. arXiv:2505.04834 confirms this as a design error
that reduces security posture. MIND must enforce the lockfile on every build
invocation without a flag, for both binaries and libraries.

### AP-3: Function coloring via async/await keywords on function signatures (Python, JavaScript, Rust async)

The function-coloring problem is that once a function is `async`, every caller
must also be `async`. Django's ORM still lacks full async support in 2025 because
retrofitting it would require rewriting every caller. Zig's retrospective
explicitly identifies this as the reason the old async system was removed. MIND
should not add `async fn` syntax; the `Scheduler` context injection pattern (§6,
recommendation 11-R1) achieves async behavior without bifurcating the function
namespace.

### AP-4: Borrow checker as the sole memory safety mechanism (Rust's ergonomic cost)

Rust's borrow checker is correct but imposes a steep ergonomic tax. Swift's
`Sendable` story in 6.0 showed how a conservative safety system can make large
codebases impossible to compile without `@unchecked` escape hatches everywhere
(a known issue documented by Sinclair, 2024). ATS's failure to achieve adoption
despite stronger guarantees confirms the pattern. MIND's memory safety model must
have a default that requires zero annotations for the common case.

### AP-5: Stackless coroutines tied to function signatures (Zig 0.10–0.11)

Zig documented four concrete problems with this approach (LLVM optimization,
debugger opacity, cancellation unsoundness, function-pointer stack-size
unknowability). The cancellation problem is particularly dangerous for governance
pipelines: a coroutine suspended mid-evidence-chain-write that is then cancelled
can leave the evidence store in an inconsistent state. Structured concurrency via
sender/receiver lifetimes (§6, recommendation 11-R2) solves this by construction:
a sender cannot be cancelled after it has been connected to a receiver that has
started consuming it.

### AP-6: Central registry as the single point of trust (crates.io, npm central registry)

Central registries are single points of failure for supply-chain attacks. The
2024 SLSA blog documents how dependency-confusion and typosquatting attacks
exploit the name-resolution ambiguity that central registries create. Git-URL
identity for MIND packages (RFC 0009 §3, AP-2 above) eliminates the attack class
at the design level. The optional MPIP index is a discovery convenience, not the
identity root.

### AP-7: Semver without a content-addressed pin (npm before lockfiles)

Semver ranges like `^1.2.3` are not content addresses. An author can yank and
re-publish a version (with different content) on most registries, and a semver
resolver may pick a newly-published higher patch version that was not present
during the last build. MIND's Phase E lockfile pins the content hash; RFC 0009's
semver ranges are resolved once and the resulting rev + tree_sha256 is written to
the lock. Once in the lock, neither the semver range nor the registry can change
what is fetched.

### AP-8: Grand unified type theory without practical adoption path (ATS/Xanadu)

ATS3 continues development as of 2024, with fewer than 200 public repositories.
The lesson is that formal correctness guarantees expressed as proof terms inside
function bodies are not adopted by working engineers. RFC 0010's memory safety
model should express safety invariants as type-level properties (effect rows,
region annotations) not as proof terms. Proof terms may be added as an optional
verification layer (analogous to Verus being a separate tool from Rust) but must
not be required for ordinary MIND programs.

## 9. Top 10 Most-Novel Ideas with MIND Adoption Proposals

### Idea 1: `ndet` effect as a first-class governance determinism marker

**Source:** Koka v3.1.0 effect system (Microsoft Research, 2024).
**Novelty:** Tracking non-determinism through the function type system prevents
governance rules from silently calling non-reproducible operations.
**MIND adoption:** Add `ndet` to the MIND effect row syntax. A function with
effect type `<pure>` cannot call any function with `<ndet>` without a `commit`
wrapper that pins the result and appends it to the evidence chain. Governance
rules are required to be `<ndet>` free in their final output.

### Idea 2: Splittable non-lexical regions (Spegion model) for async-safe arenas

**Source:** Spegion, ECOOP 2025, arXiv:2506.02182 (Hughes, Vollmer, Batty).
**Novelty:** Region lifetimes that span multiple stack frames, enforced via
effect annotations rather than borrow annotations.
**MIND adoption:** `region r { ... }` introduces a lexical region; `fn foo[r](x: &r T)` propagates the region effect. The RFC 0010 type checker verifies that no
`&r` reference escapes after `r` is freed, via effect-row tracking.

### Idea 3: Typed `Pid[T]` message protocol identifiers for MIND task handles

**Source:** NVLang, arXiv:2512.05224 (December 2025).
**Novelty:** Encode actor message protocols as sum types in process-identifier types,
catching protocol violations at compile time.
**MIND adoption:** `mind.task.Handle[M]` where `M` is the message sum type the
task accepts. `send(handle, msg)` type-checks that `msg: M`. A `Supervisor`
holding `Handle[M]` children can only send `M` messages, preventing cross-task
protocol confusion.

### Idea 4: `Io` context injection to defeat function coloring

**Source:** Zig's new async I/O design (Kristoff, 2025).
**Novelty:** I/O execution strategy is a runtime-injected parameter, not a
compile-time function attribute, so library code is execution-model-agnostic.
**MIND adoption:** Add `mind.io.Scheduler` to the MIND context. Functions that
call I/O primitives implicitly thread the context. A `BlockingScheduler`,
`ThreadPoolScheduler`, and `ReplayScheduler` are the three standard
implementations. No `async fn` syntax is added.

### Idea 5: `ReplayScheduler` for deterministic governance-pipeline testing

**Source:** Programming Journal 2022 (concurrency-model-agnostic record & replay);
arXiv:1805.06267 (deterministic R&R for actor languages).
**Novelty:** A scheduler that replays a captured trace makes any async governance
pipeline perfectly reproducible.
**MIND adoption:** `mindc test` runs async tests under `ReplayScheduler` by
default. The first run records a trace to `tests/fixtures/<test>.mind-trace`;
subsequent runs verify against it. A `--record` flag updates the trace.

### Idea 6: Go-style dual-file `Mind.toml` + `Mind.lock` with default enforcement

**Source:** arXiv:2505.04834 (lockfile design space study, 2025); Go module
system design.
**Novelty:** Default enforcement (not optional) is the single biggest driver of
lockfile adoption and the primary supply-chain security lever.
**MIND adoption:** `mindc build` fails immediately if `Mind.lock` is absent and
`--update-lock` was not passed. Lock entries are `{tree_sha256, fetch_url,
timestamp_utc}`; the format is minimal and diff-friendly.

### Idea 7: MPIP static HTTP mirror protocol for registry-free package discovery

**Source:** Go module proxy protocol design; JSR open-source registry design.
**Novelty:** A static HTTP layout over a git host allows any server to act as a
MIND package mirror without custom software.
**MIND adoption:** Define `MINDENV_PROXY` with a comma-separated URL list and
`direct` / `off` keywords. `pkg.mindlang.dev` is one conforming node. Any
Gitea/Forgejo/GitHub instance with a CI pipeline can auto-publish MPIP metadata.

### Idea 8: Generational-reference generation tags inside regions are free

**Source:** Vale's hybrid-generational memory model (Wesnoski, 2020–2024).
**Novelty:** Generation checks on non-owning pointers can be elided entirely when
the pointer is known to refer into a live region.
**MIND adoption:** The MIND compiler tracks region liveness in the type system.
Inside a `region r { ... }` block, all `&r T` dereferences emit no generation
check. Outside the block, `&r T` references that escaped become checked heap
pointers. This gives zero-cost safety inside regions and a bounded (one integer
comparison) safety cost outside.

### Idea 9: Sender/receiver algebra for composable async pipelines

**Source:** C++26 `std::execution` / P2300 (adopted June 2024, WG21 St. Louis).
**Novelty:** Async values as composable first-class objects with structured
lifetime, enabling `when_all`, `race`, and `then` without callback spaghetti.
**MIND adoption:** `async { expr }` evaluates to `Sender[T]`. `await sender` in
a scheduled context (via context injection) produces `T`. Pipelines are expressed
as `sender |> then(f) |> when_all(other_sender)`. The sender/receiver model
guarantees that all computations complete before the enclosing scope exits —
structured concurrency by construction.

### Idea 10: Exhaustive `callconv` tagged union for C ABI expression without macros

**Source:** Zig 0.14.0 `CallingConvention` redesign (released early 2025).
**Novelty:** An exhaustive enumeration of platform ABIs as a typed attribute,
with a target-aliased `.c` default, eliminates macro-based ABI dispatch.
**MIND adoption:** `callconv(.c)` is the default for `extern fn` declarations.
`callconv(.sysv)`, `callconv(.win64)`, `callconv(.aapcs)` override for specific
targets. The MIND type checker verifies that struct arguments that require
by-pointer passing in a given ABI are correctly represented in the IR, preventing
the silent miscompiles that result from incorrect struct classification.
